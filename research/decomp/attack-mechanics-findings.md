## Executive summary

- **The damage/stagger core is in good shape.** The full `zz_003cd5c_` damage formula is
  decoded and table-extracted (§ah), the gauge/stagger model is ported (gauges.ts), and —
  important negative result — the wiki's "hidden resistance/falloff" **is** the already-ported
  combo-rank falloff table `DAT_802c7ca0`. Nothing new to implement there; the risk is
  double-implementing it (see mechanic U).
- **Friendly fire is already ROM-correct in the formula and reachable in the sim loops** —
  same-team hits multiply by 0.25 unless record `flagsA & 0x1000`. What's missing is
  regression tests and the guard/block divisor wiring (mechanic V).
- **The biggest genuinely-new ROM findings of this audit**: (1) the attack-command decision
  layer (actor+0x585/+0x586 command/subtype bytes, decision testers in chunk_0009.c, command
  struct at actor+0x4b0 → actor+0x591 dispatch); (2) a complete ammo/refill system
  (3 weapon cells at +0x774/+0x77c/+0x784, refill types gradual/all-at-once/deployed-return,
  infinite-ammo rule); (3) a status-effect framework (status id +0x71a, timer +0x71c,
  immunity mask +0x5a0, shrink/grow +0x70c/+0x70e, param tier +0x74a); (4) Power Burst
  anchors (+0x6fb arm window, +0x6fc active flag: 2x out / 0.5x in / faster reload);
  (5) a real button-mash counter (+0x550, cap 4) in the attack-active handler.
- **The biggest blockers**: the physical-button → command mapping (the +0x5d4 word is a
  *transformed* button space; two corpus reads disagreed about which bit is which), the
  Power Burst meter/duration (no accumulator found in code — needs a RAM diff), the
  move → animation-bank → hitbox-record mapping (structurally decoded, per-move binding
  unknown), and projectile solidity (no generic proj-vs-proj code exists; must be per-type
  dispatch or absent).
- **What cheap agents can do now without guessing**: schema/shell work (command resolver
  types, weapon-cell ammo model, status runtime shell, burst state shell), regression tests
  for already-derived behavior (friendly fire, combo falloff, gauge stagger), and debug
  overlay fields. Everything context-numeric stays behind `BLOCKED until` gates.

### A. B/X contextual attack resolution (`contextual_attack_resolution`)

- **wiki_taxonomy_claim**: B performs a context-dependent attack (melee when close/appropriate,
  ranged otherwise); X performs the secondary/special attack; some borgs' B and X swap roles
  or share ammo.
- **current_port_behavior**: `PlayerInput` carries `attack` (B, held bool) and `special` (X,
  pressed bool) only ([types.ts:21-42](../../packages/combat/src/types.ts)). `stepAttacks`
  resolves B by iterating `["melee","shot"]` ordered by `actionProfile.primary`
  ([combat.ts:592-635](../../packages/combat/src/combat.ts)) — a per-borg static preference,
  not per-press context; X is an instant AoE burst (combat.ts:549-589). No command
  abstraction shared with AI.
- **rom_or_asset_evidence** (corpus read, this audit, except where noted):
  - Command struct: actor+0x4b0 → pointer; bytes +0/+1 (buttons) and +4 (command) copied to
    actor+0x660/+0x661/+0x591 — `FUN_800562b8` chunk_0007.c:23 and `zz_006a8c0_` @0x8006a8c0
    chunk_0009.c:978-980.
  - Decision brain: `FUN_800699d8` @0x800699d8 chunk_0009.c:220 tests transformed button word
    actor+0x5d4 in priority order via testers `zz_0069cb0_` (→ +0x585=5, requires target:
    `if (actor+0x4a0 == 0) return 0;` chunk_0009.c:412), `zz_0069c50_` (bit 0x1000 → 5),
    `zz_0069bf0_` (bit 0x400 → 3, sets +0x595=1 "charged" mark), `zz_0069b98_` (bit 0x80 → 2),
    `zz_0069b10_` (bit 0x40 → 1), `zz_0069a88_` (bit 0x20 → 1, with grounded check).
  - Subtype: `zz_0069ea4_` chunk_0009.c:500-534 / `zz_0069fe0_` :554-587 set +0x586 ∈ 0-5
    from more button bits and state (value 4 = air/elevated; 5 = charged-ranged gated on
    `FLOAT_8043762c < actor+0x6a4` :555-556; a matching gate on +0x694 at :497-498).
  - Execution: `zz_006a104_` @0x8006a104 chunk_0009.c:603 looks up (type,subtype) through a
    per-actor table at +0x4ec and triggers the action.
  - Dispatch consumption: `FUN_8005d494` (attack-active state handler) branches +0x591==1/2
    (melee family) vs 4/5/6 (ranged family) — chunk_0007.c:4750-4782.
- **confidence**: DERIVED_ROM for the pipeline structure (decision testers → type/subtype →
  per-actor action table → state handler). BLOCKED for: (1) physical-button → +0x5d4 bit
  mapping (the word is transformed; the burst search read bit 0x200 as Y while the command
  search read 0x200 as B — at least one is wrong, both are guesses over a non-raw bit space),
  (2) the writer of the +0x4b0 command struct (not found in corpus — likely input/AI
  plumbing outside the battle chunks), (3) the +0x585/+0x586 ↔ +0x591-value correspondence,
  (4) `FLOAT_8043762c`'s numeric value (not dumped yet — dumpable via `dol.py`).
- **exact_values_found**: command types {1,2,3,5}; +0x591 branch families {1,2} / {4,5,6};
  subtypes {0..5}; ranged-requires-target gate.
- **unknowns**: full decision tree with real button identities and thresholds.
- **likely_ts_touchpoints**: `packages/combat/src/types.ts` (AttackCommand type),
  new `packages/combat/src/commandResolver.ts`, `combat.ts` `stepAttacks`,
  `battle.ts` input plumbing, `packages/combat/src/ai.ts`.
- **trace_needed**: T1 (command-byte mapping), plus a `dol.py` read of FLOAT_8043762c.
- **cheap_agent_ticket_id**: ATK-001 (schema), ATK-002 (resolver tests), ATK-003 (real
  decision tree — blocked until T1).

### B. Melee priority over ranged at melee range (`melee_priority_at_range`)

- **wiki_taxonomy_claim**: at melee range, B produces the melee attack even for borgs that
  otherwise shoot.
- **current_port_behavior**: no distance check at all — `actionProfile.primary` fixes the
  order; a hybrid tries melee first only if `primary === "melee"` (combat.ts:592-609).
- **rom_or_asset_evidence**: the only distance gates found so far sit inside the
  subtype/tester functions (`FLOAT_8043762c` vs +0x694/+0x6a4, chunk_0009.c:497-498/555-556)
  and gate *charged-ranged* selection, not a general "melee wins up close" rule; ranged
  testers additionally require a live target (+0x4a0). The full priority rule is inside the
  not-yet-located command-struct writer / tester internals.
- **confidence**: partial DERIVED_ROM (distance-gated variant selection exists); BLOCKED for
  the actual B-at-melee-range rule.
- **exact_values_found**: none (threshold constant address known, value undumped).
- **unknowns**: whether melee-priority is distance-driven, target-state-driven, or per-borg.
- **likely_ts_touchpoints**: `commandResolver.ts` (once ATK-003 unblocks).
- **trace_needed**: T1 scripted distance sweep.
- **cheap_agent_ticket_id**: ATK-003.

### D–K. Contextual melee variants (`contextual_melee_variants`)

standing (D), ground dash (E), air dash (F), flying (G), stabbing (H), jumping (I),
neutral air (J), landing (K).

- **wiki_taxonomy_claim** (per search extract + brief): melee differs by movement context —
  e.g. air-dash melee = B during/after an air dash; stabbing melee = B while airborne
  directly above an enemy; landing melee = B right after landing.
- **current_port_behavior**: exactly one melee action per borg (`MeleeActionDef`) with a
  combo chain; swings are startable from any non-busy state; no movement-context variants
  (combat.ts:598-609, actionProfiles.ts:20-35).
- **rom_or_asset_evidence**:
  - Asset side (CONFIRMED_ASSET, negative): animation banks are flat (group,slot) enums with
    NO movement-context axis — group 1 holds 5-23 generic `attack_s*` slots per borg;
    no `dash_attack`/`air_attack`/`landing` labels exist anywhere in the pipeline
    (`research/format-specs/borg-animation-banks.md`,
    `research/animation-actions/borg-action-assets.md`,
    `scripts/validate-borg-animation-actions.mjs:34-78`).
  - ROM side (corpus read, this audit): the command subtype byte +0x586 (0-5) IS a
    context/variant channel — value 4 is set on an air/elevated gate, testers apply grounded
    checks (chunk_0009.c:273-316, 500-534); +0x1d0f is a per-attack direction/variant
    selector 0-5 consumed in the attack handlers (chunk_0007.c:4259-4813); command values
    1 vs 2 are two distinct melee families.
  - Hitbox side (CONFIRMED_ASSET): each borg has up to 64 frame-windowed attack hitbox
    records selected via remap-B slots by spawner "kind" (`hit-bin-format.md`,
    `zz_008ac80_` @0x8008ac80) — enough authored data to back per-context hitboxes.
- **confidence**: DERIVED_ROM that a variant/context channel exists (type 1/2 + subtype 0-5 +
  +0x1d0f); WIKI_TAXONOMY_ONLY for the specific 8-context list and which context maps to
  which variant; CONFIRMED_ASSET for the hitbox/animation containers.
- **exact_values_found**: subtype range 0-5; variant selector range 0-5; per-borg attack-bank
  counts (e.g. pl0615: 23 group-1 slots).
- **unknowns**: context → (command,subtype) → (bank, remap slot) mapping.
- **likely_ts_touchpoints**: `types.ts` (movement-context enum on BorgRuntime),
  `movement.ts` (context capture: dashing/air/landing frames already computable),
  `commandResolver.ts`, `actionProfiles.ts` (per-context profile slots).
- **trace_needed**: T2 (variant table), after T1.
- **cheap_agent_ticket_id**: ATK-004 (context enum + capture), ATK-005 (profile mapping,
  schema only until T2).

- **likely_ts_touchpoints**: none yet (scaffolding only).

### O. Penetration (`penetration`)

- **wiki_taxonomy_claim**: certain attacks penetrate certain objects; melee and explosions
  effectively have total penetration.
- **current_port_behavior**: every projectile is consumed by its first valid borg hit
  (`consumed = true; break;` combat.ts:936-939). Melee already "penetrates" trivially
  (per-swing hit list `meleeHitUids`, one hit per target per swing, combat.ts:654-692).
- **rom_or_asset_evidence** (corpus read, this audit):
  - No destroy-projectile-on-hit exists in the collision/hit-resolution code; object death is
    `object+0x10 = 2` set by actor-gated logic (chunk_0013.c:1188, gated on an actor flag
    +0x281), i.e. persistence-after-hit is the structural default and despawn is per-type.
  - Re-hit pacing: damage record s8+0x16 (`rehitIntervalFrames`) reloads a per-OBJECT counter
    at object+0x4e; on expiry the hit re-applies (chunk_0013.c:1175-1182). Counter is global
    to the object, not per-target.
  - Collision gate: hit allowed iff `(attackerRec.flagsB & 0x4000)==0 ||
    (targetRec.flagsB & 0x2000)!=0` (chunk_0003.c:7080). NOTE: flagsB bit 0x4000 is ALSO the
    skip-combo-falloff bit in the damage formula (§ah step 16) — one bit observed in two
    consumers; do not treat the collision-gate reading as settled semantics.
  - All 9 extracted borg-family damage records have `rehitIntervalFrames = 0` and flagsB = 0
    (`packages/combat/src/data/damageRecords.json`); the interesting values live in the
    generic/comhit table `DAT_802c4760` and per-move hit-bin data (partially extracted:
    `research/decomp/data/damage-records-802d46e0.json` covers both tables' bytes).
- **confidence**: DERIVED_ROM (mechanisms: rehit counter, actor-gated despawn, flag gate);
  BLOCKED (per-move values and true flag semantics).
- **exact_values_found**: gate expression above; record field offsets.
- **unknowns**: which moves persist through borgs; explosion spawn-on-impact (no effect-id
  consumer found in the generic hit path — corpus-read negative).
- **likely_ts_touchpoints**: `types.ts` `Projectile` (penetration/rehit fields),
  `combat.ts` `stepProjectiles`.
- **trace_needed**: T6.
- **cheap_agent_ticket_id**: ATK-008 (consumption refactor — blocked until flag mapping/T6).
